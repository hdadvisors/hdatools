## What to build

A single self-contained HTML dashboard, opened locally from disk (`file://`),
regenerated by a Python script that parses the repo. Nothing is hosted;
nothing loads from a CDN or network at view time.

### Architecture (settled decisions)

- **Generator:** one Python script, `plans/dashboard/generate_dashboard.py`,
  run with the system Python (3.13, on PATH; fallback
  `C:\Python313\python.exe`). Prefer stdlib only. For rendering markdown to
  HTML, either a small vendored/inlined pure-Python renderer or an inlined JS
  renderer inside the HTML is fine — no pip installs unless truly necessary,
  and no CDN references ever.
- **Output:** `plans/dashboard/dashboard.html` — **gitignored** (add the
  ignore rule). The generator and any template/assets are committed. `plans/`
  is already in `.Rbuildignore`, so nothing here touches `R CMD check`.
- **Data sources — parse directly, no separate status file:**
  - `plans/ROADMAP.md` phase table + cross-repo ledger
  - `plans/DECISIONS.md` settled + open tables
  - the active phase plan's header table, session headings, and findings
    (note: phase plans are drafted just-in-time, so the current phase may have
    **no plan file yet** — handle that gracefully)
  - `plans/archive/*.md` completion headers
  - `DESCRIPTION` (Version), `NEWS.md` (dev bullets + release headings)
  - local git: branches, ahead/behind vs main, merge points, tags, recent log
  - `gh` CLI for PR list/status and latest CI runs — **with offline
    fallback**: if `gh` fails or there's no network, render everything else
    and stamp the PR/CI panel "last fetched <timestamp>" using cached data
    (cache the last successful `gh` JSON next to the generator, also
    gitignored)
- **Update flow:** Claude-session convention. Add a short section to the
  project `CLAUDE.md` instructing sessions to re-run the generator after
  commits that change project state, and document the manual path. Also
  create `plans/dashboard/update-dashboard.bat` that regenerates and opens
  the dashboard in the default browser (for terminal/double-click use; I'm
  the only user, no .vbs needed).

### Layout and content (settled decisions)

Single page, **tab bar** across the top, utilitarian styling — plain,
readable, fast; no framework, no animation flourishes. Five tabs:

1. **Roadmap** — the phase table as a visual: each phase with target release,
   branch, status, scope summary; the active phase expanded to show its
   sessions (done / in progress / not started, from the plan file's structure
   and findings) and exit criteria. Show current `DESCRIPTION` version
   prominently.
2. **Branches & PRs** — a branch/merge diagram (main, phase branches, merge
   commits, version tags) **annotated as a learning aid**: plain-language
   notes on what each branch is for, its PR state, ahead/behind counts, and
   what happens next (e.g. "phase branch — PR opens when the phase's release
   checklist passes"). Scope: **modernization era only** (2026-07-16 onward,
   i.e. commit `5963754` / the design review forward). The 2022–23 legacy
   branches (`jtk`, `category-colors`, `showtext-knitr-fix`, etc.) go in a
   collapsed "pre-modernization history" footnote, not the diagram.
3. **Decisions** — DECISIONS.md rendered as browsable tables: settled
   decisions (date, ref, decision, rationale, binds) and open questions
   grouped by the phase gate that settles them.
4. **Release & CI** — current version and NEWS dev bullets; latest CI run
   status per active branch (from `gh`, with the offline fallback above);
   release-checklist steps for the active phase; the cross-repo consumer
   ledger from ROADMAP.md.
5. **Docs** — rendered views of `CLAUDE.md`, `README.md`, and `NEWS.md` so I
   can read them without opening separate windows. Regenerated fresh each
   run, sub-tabs or a simple switcher within the tab.

### Nice-to-have (include if cheap, skip if it bloats the plan)

- **Consistency warnings:** the generator cross-checks derived git/gh facts
  against the plan docs and flags mismatches at the top of the dashboard —
  e.g. "ROADMAP table says Phase 1 'ready' but v0.3.0 tag exists" or "phase
  plan not yet archived." The plan docs have lagged reality before; a nudge
  to fix the docs (never auto-edit them) fits the dashboard's purpose.

### Constraints and conventions

- Work on the existing `dashboard-tooling` branch; when done, its own small
  PR to main. This is repo tooling, not package code — it must not touch `R/`, `man/`,
  `NAMESPACE`, `DESCRIPTION`, or `docs/`, and `devtools::check()` must be
  unaffected (verify `.Rbuildignore` covers everything you add).
- Windows: use pathlib, write files as UTF-8 explicitly, and remember `.bat`
  quoting rules. Don't shell out to R at all — the generator needs only
  Python, git, and (optionally) gh.
- Parsing is coupled to the markdown conventions in `plans/` — keep parsers
  tolerant (anchor on table headers and `## ` headings, not line numbers),
  and fail soft: a section that fails to parse renders as an inline warning
  in its panel, never a crashed generator.
- Verify end-to-end before finishing: run the generator, open the HTML in
  the browser pane, and check every tab renders with real repo data — including
  the offline path (simulate `gh` failure) and the no-plan-file-yet case for
  the next phase.
- Docs are first-class: update project CLAUDE.md (regeneration convention +
  a one-line pointer to `plans/dashboard/`) in the same session. Suggest a
  commit message at the end; no Claude/Anthropic co-author line.

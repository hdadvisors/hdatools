## Session 2 — systemfonts/ragg migration

**Goal:** replace the `showtext`/`sysfonts` font stack with `systemfonts` + `ragg`,
and remove the `.onLoad` global side effects.

**Steps:**

1. In `DESCRIPTION`: replace `showtext` and `sysfonts` in `Imports` with
   `systemfonts` and `ragg`; add `textshaping` to `Imports` if font shaping
   coverage requires it (verify with `systemfonts::system_fonts()` on the bundled
   TTFs first).
2. Rewrite `register_hda_fonts()` (currently in `R/zzz.R` or `R/fonts.R`):
   - Replace `sysfonts::font_add()` + `showtext::showtext_auto()` calls with
     `systemfonts::add_font()` / `register_variant()` for each bundled TTF.
   - Remove `knitr::opts_chunk$set(fig.showtext = TRUE)` — this global side effect
     is the main user-visible difference; note it in NEWS.
   - Keep the `hdatools.fonts` option guard (`getOption("hdatools.fonts", TRUE)`)
     so callers can opt out of auto-registration.
3. Audit `R/zzz.R`: if `.onLoad` still calls `register_hda_fonts()` automatically,
   keep that pattern but confirm it no longer sets knitr globals.
4. Verify each bundled TTF file (in `inst/fonts/` or similar) is still referenced
   correctly by the new `systemfonts` paths.
5. Add or update a `dev: ragg_png` note in `README.md` — consumers must add this
   to their `_quarto.yml` for fonts to render; without it Quarto uses the default
   Cairo device and the registration has no effect.
6. `devtools::document()`, `devtools::test()`. Expect font-related test churn if
   any test directly exercised `showtext`; fix tests, don't skip them.
7. NEWS.md bullet covering: (a) showtext/sysfonts dropped, (b) consumers must add
   `dev: ragg_png`, (c) knitr global option no longer set by the package.

**Verification:** `devtools::check()` at 0 errors / 0 warnings / license-NOTE-only;
`systemfonts::system_fonts()` shows the bundled faces after `library(hdatools)`;
a `ggplot2` PNG rendered via ragg (`ragg::agg_png()`) uses the bundled typeface
without warnings.

**Stop here if:** `systemfonts::add_font()` / `register_variant()` can't locate
one of the bundled TTFs — resolve the path before proceeding, as the comparison
in Session 3 depends on fonts actually loading.

## Session 3 — Consumer render comparison (fhfh) + guidance docs

**Goal:** validate that the systemfonts/ragg text metrics are acceptable in the
binding consumer (fhfh), and publish the `dev: ragg_png` guidance.

**Steps:**

1. In the **fhfh** repo session (not this one): install the `release-0.5.0` branch
   of hdatools, add `dev: ragg_png` to fhfh's `_quarto.yml`, and re-render a
   representative subset of charts.
2. Produce a side-by-side comparison (0.4.0 output vs 0.5.0 output) for at least
   one chart per brand used by fhfh. Share with Jonathan for sign-off before
   tagging.
3. Back in this repo: add a `vignettes/articles/ragg-migration.Rmd` (or a section
   in the existing setup article) explaining:
   - Why the switch was made (showtext guidance, PDF text quality).
   - Exact `_quarto.yml` block to add (`knitr: opts_chunk: dev: "ragg_png"`).
   - What changes visually (minor metric shift, richer hinting) and what doesn't
     (colors, theme layout, opt-out path via `options(hdatools.fonts = FALSE)`).
4. Update `get_output_format()` to recognize `"typst"` and `"docx"` as distinct
   formats rather than mapping them to `"pdf"` (§4 of the design review; cheap
   fix, right moment since we're already touching rendering infrastructure).
5. `devtools::document()`, `devtools::check()`.

**Verification:** fhfh side-by-side signed off by Jonathan; `get_output_format()`
returns `"typst"` for a typst knitr context; guidance article renders without
errors via `pkgdown::build_articles()`.

**Stop here if:** fhfh comparison reveals metric shifts bad enough that charts
require layout fixes in fhfh — pause, scope the fhfh changes, and get Jonathan's
go/no-go before tagging 0.5.0.
